## 阶段摘要

本阶段将插件体系设计审计转为可执行实施账本。目标不是重新设计一套大型 marketplace，也不是把 Devtools、Logger 这类宿主基础设施整体插件化，而是把现有 `PluginManifest` / `PluginManager` 从“基础动态加载能力”推进到“可被用户安全启用、禁用、信任、诊断和扩展的产品级插件架构”。

实施优先级遵循四个原则：

1. 先修安全边界，再扩能力面。
2. 先保证 diagnostics 与真实 runtime 一致，再做 UI/CLI 体验。
3. 所有动态代码加载必须经过显式 trust，不把 `tools` 当作天然安全权限。
4. `tools/skills/MCP/hooks/commands/agents` 统一走 capability lifecycle，不再让 manifest 声明和实际加载脱节。
5. Devtools host、Logger core、Permission audit log 保持宿主内核能力；插件只能贡献受 trust/permission 管控的 panel、provider、action、sink、formatter。
6. Claude Code 兼容采用导入/适配层，不承诺直接运行 Claude Code 插件 runtime。

## 里程碑计划

### Milestone A：安全边界先行

目标：修复“未信任动态代码执行”和路径逃逸风险。

- PLUGIN-01 Trust gate 收紧
- PLUGIN-05 安全 module resolver 与发现模型修复

完成标准：

- 未 trust 的 tools-only 插件不会执行模块顶层代码。
- symlink escape 被拒绝。
- root 本身作为插件目录和 root 下多插件目录两种发现模式都可用。

### Milestone B：产品状态闭环

目标：让 CLI/App 使用同一份插件状态，避免管理命令跨进程失效。

- PLUGIN-02 持久化 trusted/disabled 状态
- PLUGIN-06 CLI diagnostics 展示增强中的状态展示部分

完成标准：

- `vitamin plugin disable <id>` 对下一次 `vitamin` 启动生效。
- CLI list 能展示 trusted/disabled/loaded/errors。
- 状态文件不污染第三方插件 manifest。

### Milestone C：App runtime 能力补齐

目标：让 VCCG-09 已声明的 hook 能在默认 App 路径真实可用，并让 diagnostics 诚实。

- PLUGIN-03 App hookRegistry 接入
- PLUGIN-06 commands/agents diagnostics 真实化

完成标准：

- App start/stop 集成测试覆盖 plugin hook 注册与卸载。
- commands/agents 没有执行 adapter 时不再标记 loaded。

### Milestone D：Capability lifecycle 一体化

目标：把 `tools/skills/MCP/hooks/commands/agents` 从分散实现收敛成统一 lifecycle。

- PLUGIN-04 统一 plugin capability lifecycle adapter

完成标准：

- Skill/MCP 插件 trust 后能通过 adapter 加载。
- disable/unload 能调用 Skill/MCP 反向 adapter。
- 所有 lifecycle step 都有 loaded/skipped/disabled/error 证据。

### Milestone E：生态兼容导入

目标：兼容 Claude Code 插件生态的可迁移资产，但不直接运行其 runtime。

- PLUGIN-07 Claude Code 插件兼容导入

完成标准：

- Claude Code skill、MCP server、slash command 能转换为 Vitamin manifest。
- 转换报告明确 unsupported 字段和需要 trust 的动态能力。
- 导入后的能力仍走 Vitamin trust gate、state store 和 lifecycle adapter。

### Milestone F：宿主基础设施扩展点

目标：把 Devtools 和日志做成插件贡献点，而不是把宿主基础设施本体插件化。

- PLUGIN-08 Devtools 插件贡献点
- PLUGIN-09 日志插件贡献点

完成标准：

- Devtools server/worker/lifecycle 仍由 VitaminApp 管理。
- 可信插件可贡献 Devtools panel/provider/action。
- Logger core 和 permission audit log 不可被插件替换。
- 插件 log sink 只能接收脱敏后的事件，且 network/file sink 受 permission 控制。

## 实施顺序

1. PLUGIN-01：先阻断未 trust 动态代码执行，降低最大安全风险。
2. PLUGIN-05：紧接修复 module resolver 和 discovery 语义，避免后续能力复用不安全 resolver。
3. PLUGIN-03：补 App hookRegistry 接入，修正已声明能力在产品路径不可用的问题。
4. PLUGIN-06：先把 commands/agents diagnostics 改诚实，再考虑执行入口。
5. PLUGIN-02：引入持久化状态并改 CLI/App 初始化顺序。
6. PLUGIN-04：在状态和安全边界稳定后，统一 Skill/MCP/Hook/Command/Agent lifecycle adapter。
7. PLUGIN-07：在 lifecycle 和 trust/state 稳定后做 Claude Code 兼容导入，避免导入能力绕过 Vitamin 安全模型。
8. PLUGIN-08：在 capability adapter 成熟后接 Devtools contribution registry。
9. PLUGIN-09：最后接日志 sink/formatter 扩展，确保 redaction、audit log 和 permission 规则已稳定。
